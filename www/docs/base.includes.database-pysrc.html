 68    """ 
 69     Allow the use of different database servers using the same code base. 
 70     
 71     Drupal provides a slim database abstraction layer to provide developers with 
 72     the ability to support multiple database servers easily. The intent of this 
 73     layer is to preserve the syntax and power of SQL as much as possible, while 
 74     letting Drupal control the pieces of queries that need to be written 
 75     differently for different servers and provide basic security checks. 
 76     
 77     Most Drupal database queries are performed by a call to db_query() or 
 78     db_query_range(). Module authors should also consider using pager_query() for 
 79     queries that return results that need to be presented on multiple pages, and 
 80     tablesort_sql() for generating appropriate queries for sortable tables. 
 81     
 82     For example, one might wish to return a list of the most recent 10 nodes 
 83     authored by a given user. Instead of directly issuing the SQL query 
 84     @code 
 85       SELECT n.title, n.body, n.created FROM node n WHERE n.uid = uid LIMIT 0, 10; 
 86     @endcode 
 87     one would instead call the Drupal functions: 
 88     @code 
 89       result = db_query_range('SELECT n.title, n.body, n.created 
 90         FROM {node} n WHERE n.uid = %d', uid, 0, 10); 
 91       while (node = db_fetch_object(result)) { 
 92         // Perform operations on node->body, etc. here. 
 93       } 
 94     @endcode 
 95     Curly braces are used around "node" to provide table prefixing via 
 96     db_prefix_tables(). The explicit use of a user ID is pulled out into an 
 97     argument passed to db_query() so that SQL injection attacks from user input 
 98     can be caught and nullified. The LIMIT syntax varies between database servers, 
 99     so that is abstracted into db_query_range() arguments. Finally, note the 
100     common pattern of iterating over the result set using db_fetch_object(). 
101     
102     Perform an SQL query and return success or failure. 
103     
104     @param sql 
105       A string containing a complete SQL query.  %-substitution 
106       parameters are not supported. 
107     @return 
108       An array containing the keys: 
109          success: a boolean indicating whether the query succeeded 
110          query: the SQL query executed, passed through check_plain() 
111    """ 
112    result = db_query(sql, true); 
113    return {'success' : (result != False), 'query' : check_plain(sql)};
