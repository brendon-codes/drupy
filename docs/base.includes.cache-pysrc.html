 82    """ 
 83     Store data in the persistent cache. 
 84     
 85     The persistent cache is split up into four database 
 86     tables. Contributed modules can add additional tables. 
 87     
 88     'cache_page': This table stores generated pages for anonymous 
 89     users. This is the only table affected by the page cache setting on 
 90     the administrator panel. 
 91     
 92     'cache_menu': Stores the cachable part of the users' menus. 
 93     
 94     'cache_filter': Stores filtered pieces of content. This table is 
 95     periodically cleared of stale entries by cron. 
 96     
 97     'cache': Generic cache storage table. 
 98     
 99     The reasons for having several tables are as follows: 
100     
101     - smaller tables allow for faster selects and inserts 
102     - we try to put fast changing cache items and rather static 
103       ones into different tables. The effect is that only the fast 
104       changing tables will need a lot of writes to disk. The more 
105       static tables will also be better cachable with MySQL's query cache 
106     
107     @param cid 
108       The cache ID of the data to store. 
109     @param data 
110       The data to store in the cache. Complex data types will be automatically serialized before insertion. 
111       Strings will be stored as plain text and not serialized. 
112     @param table 
113       The table table to store the data in. Valid core values are 'cache_filter', 
114       'cache_menu', 'cache_page', or 'cache'. 
115     @param expire 
116       One of the following values: 
117       - CACHE_PERMANENT: Indicates that the item should never be removed unless 
118         explicitly told to using cache_clear_all() with a cache ID. 
119       - CACHE_TEMPORARY: Indicates that the item should be removed at the next 
120         general cache wipe. 
121       - A Unix timestamp: Indicates that the item should be kept at least until 
122         the given time, after which it behaves like CACHE_TEMPORARY. 
123     @param headers
